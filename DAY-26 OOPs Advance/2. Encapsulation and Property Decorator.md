# 2. Encapsulation and Property Decorator

Encapsulation is the practice of bundling data (attributes) and behavior (methods) inside a single class, while restricting direct access to some of the object's internal components. This prevents accidental data corruption and enforces security boundaries.

---

## 1. Access Modifiers: Public, Protected, and Private

Unlike languages like Java or C++ which have strict keywords (`public`, `private`), Python uses naming conventions to define the visibility of class members.

```text
  Visibility Modifiers in Python:
  
  ┌────────────────────────────────────────────────────────┐
  │ Public: Default access. Available everywhere.          │
  │   ┌────────────────────────────────────────────────────┤
  │   │ Protected (_name): Access inside class & subclasses.│
  │   │   ┌────────────────────────────────────────────────┤
  │   │   │ Private (__name): Strictly class-restricted.   │
  └───┴───┴────────────────────────────────────────────────┘
```

### A. Public Members
All attributes and methods are public by default. They can be accessed from inside the class, by subclasses, or by external code directly.

**Input:**
```python
class Account:
    def __init__(self, owner, balance):
        self.owner = owner      # Public attribute
        self.balance = balance  # Public attribute

acc = Account("Alice", 5000)
print(acc.owner, "has", acc.balance)  # Accessing directly
acc.balance = -1000  # Modifying directly (dangerous!)
print("New Balance:", acc.balance)
```
**Output:**
```text
Alice has 5000
New Balance: -1000
```

### B. Protected Members (`_`)
A protected member is prefixed with a **single underscore** `_`. 
- **The Convention:** This is a gentle warning to other developers: "This is internal state. Do not modify or read it directly from outside the class or its subclasses."
- **Note:** Python does **not** enforce this at runtime. You can still read/write it from the outside, but doing so violates Pythonic best practices.

**Input:**
```python
class Account:
    def __init__(self, owner, balance):
        self.owner = owner
        self._routing_number = 121000248  # Protected attribute

acc = Account("Alice", 5000)
print("Routing Number:", acc._routing_number)  # Python allows it, but it's bad practice!
```
**Output:**
```text
Routing Number: 121000248
```

### C. Private Members (`__`)
A private member is prefixed with a **double underscore** `__`.
- **Strict Restriction:** These attributes and methods cannot be accessed or modified directly from outside the class. Trying to do so raises an `AttributeError`.

**Input:**
```python
class SecureAccount:
    def __init__(self, owner, pin):
        self.owner = owner
        self.__pin = pin  # Private attribute

acc = SecureAccount("Bob", 4321)
print(acc.owner)

# Trying to access __pin directly will crash!
try:
    print(acc.__pin)
except AttributeError as e:
    print("Access Error:", e)
```
**Output:**
```text
Bob
Access Error: 'SecureAccount' object has no attribute '__pin'
```

---

## 2. Name Mangling under the hood

How does Python enforce privacy if it doesn't have compiler-level access security? It uses a mechanism called **Name Mangling**.

Whenever Python sees a double underscore prefix like `__pin` inside a class named `SecureAccount`, it internally rewrites the attribute name to `_SecureAccount__pin` (adding `_ClassName` before it). 

This means the variable still exists under this mangled name, and can technically be accessed, although you should never do so in production code.

**Input:**
```python
class SecureAccount:
    def __init__(self, owner, pin):
        self.owner = owner
        self.__pin = pin

acc = SecureAccount("Bob", 4321)

# Accessing via Name Mangling:
print("Mangled PIN access:", acc._SecureAccount__pin)
```
**Output:**
```text
Mangled PIN access: 4321
```

## 3. The `@property` Decorator (Getters, Setters, Deleters)

If we make attributes private to protect them, how can external code read or modify them safely? We use **Getters** and **Setters**.

In Python, the most elegant and clean way to implement getters and setters is by using the built-in `@property` decorator. This allows you to access methods *as if they were standard attributes* (without parentheses), while running validation logic behind the scenes.

Let's build a safe temperature tracker where temperature cannot be set below absolute zero (-273.15°C).

**Input:**
```python
class TemperatureSensor:
    def __init__(self, celsius=0):
        self.__celsius = celsius  # Private attribute

    # 1. Getter: accessed like 'obj.celsius'
    @property
    def celsius(self):
        print("🔍 Reading temperature sensor...")
        return self.__celsius

    # 2. Setter: accessed like 'obj.celsius = value'
    @celsius.setter
    def celsius(self, value):
        if value < -273.15:
            raise ValueError("❌ Temperature cannot be below Absolute Zero (-273.15°C)!")
        print(f"🌡️ Adjusting temperature to {value}°C")
        self.__celsius = value

    # 3. Deleter: accessed like 'del obj.celsius'
    @celsius.deleter
    def celsius(self):
        print("🗑️ Resetting temperature sensor data...")
        self.__celsius = 0

# Test the sensor
sensor = TemperatureSensor(25)

# Call Getter (Notice: No parentheses!)
print("Current Temp:", sensor.celsius)

# Call Setter
sensor.celsius = 30
print("New Temp    :", sensor.celsius)

# Test validation
try:
    sensor.celsius = -300
except ValueError as e:
    print("Error:", e)

# Call Deleter
del sensor.celsius
print("After Reset :", sensor.celsius)
```
**Output:**
```text
🔍 Reading temperature sensor...
Current Temp: 25
🌡️ Adjusting temperature to 30°C
🔍 Reading temperature sensor...
New Temp    : 30
Error: ❌ Temperature cannot be below Absolute Zero (-273.15°C)!
🗑️ Resetting temperature sensor data...
🔍 Reading temperature sensor...
After Reset : 0
```

---

## 4. Why Use `@property`?
- **Backward Compatibility:** You can start with simple public attributes, and later convert them to properties with validation *without changing the calling code* (which remains `obj.attribute`).
- **Data Validation:** Ensures object states stay consistent (e.g., preventing negative account balances or invalid names).
- **Read-Only Attributes:** You can create a getter without a setter, making the attribute read-only from the outside.
