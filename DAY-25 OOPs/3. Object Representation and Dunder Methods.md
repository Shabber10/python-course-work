# 3. Object Representation and Dunder Methods

Python objects have several built-in hidden behaviors. These behaviors are controlled by special methods called **Dunder Methods** (short for **D**ouble **Under**score), also known as **Magic Methods**. 

These methods are prefixed and suffixed by double underscores (e.g., `__init__`, `__str__`). You rarely call them directly; instead, Python triggers them automatically in response to specific operations.

---

## 1. The Default Object Representation
By default, if you try to print a custom object in Python, it outputs a cryptic string showing the class name and its memory location. This is not helpful for developers or users!

**Input:**
```python
class Product:
    def __init__(self, name, price):
        self.name = name
        self.price = price

item = Product("Laptop", 999.99)
print(item)
```
**Output:**
```text
<__main__.Product object at 0x0000021A45AB3F40>
```

To customize how our objects are displayed to users and developers, we override the `__str__` and `__repr__` dunder methods.

---

## 2. The `__str__` Method
The `__str__` method returns an informal, user-friendly string representation of the object. It is intended for the end-user (e.g., displaying product info in a shopping cart list).
- **Triggered By:** `print(obj)`, `str(obj)`, or formatting with f-strings `f"{obj}"`.

**Input:**
```python
class Product:
    def __init__(self, name, price):
        self.name = name
        self.price = price

    # Informal string representation for the user
    def __str__(self):
        return f"🛒 {self.name} - ${self.price:.2f}"

item = Product("Laptop", 999.99)

# This now calls __str__ automatically!
print(item)
print(str(item))
```
**Output:**
```text
🛒 Laptop - $999.99
🛒 Laptop - $999.99
```

---

## 3. The `__repr__` Method
The `__repr__` method returns an unambiguous, formal string representation of the object. It is intended for developers, debugging, and logging. 
- **Goal:** Ideally, the string returned by `__repr__` should look like a valid Python expression that can be copied and pasted to recreate the exact same object.
- **Triggered By:** `repr(obj)`, interactive REPL inspection, and when objects are nested inside collections (like lists).

**Input:**
```python
class Product:
    def __init__(self, name, price):
        self.name = name
        self.price = price

    # Formal string representation for developers
    def __repr__(self):
        return f"Product(name='{self.name}', price={self.price})"

item = Product("Laptop", 999.99)

# Get formal representation
print("Developer Repr:", repr(item))
```
**Output:**
```text
Developer Repr: Product(name='Laptop', price=999.99)
```

---

## 4. ⚠️ CRITICAL GOTCHA: Objects Inside Collections
A very common Python gotcha occurs when you put objects inside a list, tuple, or dictionary. 

Even if you have defined a beautiful `__str__` method, Python will **default to calling `__repr__`** when printing the collection! If you don't define `__repr__`, you will get a list of unreadable memory addresses.

Let's see this in action:

**Input:**
```python
class ItemWithoutRepr:
    def __init__(self, name):
        self.name = name
    def __str__(self):
        return self.name

class ItemWithBoth:
    def __init__(self, name):
        self.name = name
    def __str__(self):
        return self.name
    def __repr__(self):
        return f"Item({self.name!r})"  # !r formats it with quotes (representation)

# 1. Printing a shopping cart of items WITHOUT __repr__
cart_a = [ItemWithoutRepr("Laptop"), ItemWithoutRepr("Mouse")]
print("Cart A (No Repr):", cart_a)

# 2. Printing a shopping cart of items WITH __repr__
cart_b = [ItemWithBoth("Laptop"), ItemWithBoth("Mouse")]
print("Cart B (With Repr):", cart_b)
```
**Output:**
```text
Cart A (No Repr): [<__main__.ItemWithoutRepr object at 0x...>, <__main__.ItemWithoutRepr object at 0x...>]
Cart B (With Repr): [Item('Laptop'), Item('Mouse')]
```

> [!TIP]
> **Best Practice:** Always define at least `__repr__` for your classes. If a class has a `__repr__` but no `__str__`, Python will fall back to using `__repr__` when you print the object or call `str()`.

---

## 5. Summary: `__str__` vs. `__repr__`

| Feature | `__str__` | `__repr__` |
| :--- | :--- | :--- |
| **Purpose** | Readable, friendly representation for end-users. | Unambiguous representation for developers/debugging. |
| **Target Audience** | End-users / Shopping Cart UI. | Developers / Logs / Debuggers. |
| **Fallback** | If missing, Python calls `__repr__`. | If missing, Python uses default memory reference. |
| **Collections behavior** | Not used when printing lists/dicts. | Always used when printing lists/dicts of objects. |

## 6. Object Lifecycles: Destructors (`__del__`)

In Python, the lifecycle of an object is managed automatically:
1. **Creation (`__init__`):** Runs automatically when a new object is created to initialize its state.
2. **Destruction (`__del__`):** Runs automatically when an object is about to be destroyed (i.e., when all references to it are deleted, or the program ends). This acts as a **destructor** and is typically used for cleaning up resources (closing network connections, saving file states).

**Input:**
```python
class ResourceConnection:
    def __init__(self, resource_name):
        self.resource_name = resource_name
        print(f"🔌 Connection opened for: {self.resource_name}")

    # Destructor method
    def __del__(self):
        print(f"🗑️ Connection closed and memory cleaned up for: {self.resource_name}")

# 1. Create and destroy object using del keyword
conn = ResourceConnection("Database_Service")
del conn  # Manually trigger destruction

print("--- End of Demo ---")
```
**Output:**
```text
🔌 Connection opened for: Database_Service
🗑️ Connection closed and memory cleaned up for: Database_Service
--- End of Demo ---
```
