# 1. Introduction to OOP (Classes and Objects)

Object-Oriented Programming (OOP) is one of the most powerful and widely used programming paradigms in modern software development. Instead of writing code as a series of sequential instructions, OOP allows you to structure your programs by modeling real-world objects and their behaviors.

---

## 1. What is OOP?

Imagine you are playing a multiplayer action game like **Free Fire** or **PUBG**. The game features multiple characters: **Kelly**, **Alok**, **Hayato**, and others. 

Each character has:
- **State (Attributes/Data):** Health points (HP), movement speed, active weapon, and skills.
- **Behaviors (Methods/Actions):** Running, jumping, reloading, and shooting.

In programming terms:
- The general layout or blueprint of what a character is and does is called a **Class**.
- Each individual, active player character running around in the map is an **Object** (or an **Instance** of that Class).

Object-Oriented Programming is a method of designing programs by grouping related variables (state) and functions (behavior) into single entities called **Objects**.

---

## 2. Why Do We Need OOP? (Procedural vs. OOP)

To see why OOP is so valuable, let’s look at how we would build a multiplayer game lobby using **Procedural Programming** (the style we have been using so far) versus **Object-Oriented Programming**.

### A. The Procedural Approach (Loose Data)
In procedural programming, data is separated from functions. We might track characters using parallel lists:

**Input:**
```python
player_names = ["Alpha", "Beta", "Gamma"]
player_hps = [100, 85, 90]
player_characters = ["Alok", "Kelly", "Hayato"]

def take_damage(player_index, amount):
    player_hps[player_index] -= amount
    if player_hps[player_index] <= 0:
        player_hps[player_index] = 0
        print(f"💀 Player {player_names[player_index]} was knocked out!")
    else:
        print(f"💥 Player {player_names[player_index]} took damage! HP: {player_hps[player_index]}")

# Simulate damage
take_damage(1, 90)
```
**Output:**
```text
💀 Player Beta was knocked out!
```

> [!CAUTION]
> **Why this fails at scale:**
> - If Player Beta leaves the game, you must delete their data from *every single list* and adjust all indices. If you make a mistake, Alpha might inherit Beta's health bar!
> - The data is completely unprotected. Any part of the program can write `player_hps[0] = -500`, breaking game logic.

### B. The OOP Approach (Bundled Entity)
In OOP, we bind the player's data and actions together. If a player leaves, we simply delete that single `Player` object.

**Input:**
```python
class Player:
    def __init__(self, username, char_type, hp=100):
        self.username = username
        self.char_type = char_type
        self.hp = hp

    def take_damage(self, amount):
        self.hp -= amount
        if self.hp <= 0:
            self.hp = 0
            print(f"💀 Player {self.username} ({self.char_type}) was knocked out!")
        else:
            print(f"💥 Player {self.username} took {amount} damage! HP remaining: {self.hp}")

# Create players (Objects)
player1 = Player("Alpha", "Alok")
player2 = Player("Beta", "Kelly")

# Simulate damage directly on the object
player2.take_damage(90)
player2.take_damage(20)
```
**Output:**
```text
💥 Player Beta took 90 damage! HP remaining: 10
💀 Player Beta (Kelly) was knocked out!
```

## 3. The Four Pillars of OOP

OOP is defined by four core principles (features) that make software modular, clean, and extensible.

```text
       ┌─────────────────────────────────────────────────────────┐
       │                 THE FOUR PILLARS OF OOP                 │
       └─────────────────────────────────────────────────────────┘
            │               │               │               │
     ┌──────┴──────┐ ┌──────┴──────┐ ┌──────┴──────┐ ┌──────┴──────┐
     │ Abstraction │ │Encapsulation│ │ Inheritance │ │Polymorphism │
     └─────────────┘ └─────────────┘ └─────────────┘ └─────────────┘
```

### 1. Abstraction (Hiding Complexity)
Abstraction hides complex background details and only exposes the essential controls.
- **Real-World Example:** In *Free Fire*, you click the "Fire" button on your screen to shoot. You don't need to see the complex server-ping checks, hitbox calculations, physics trajectories, or frame rendering algorithms running in the background. The game hides that complexity and provides a simple button.

### 2. Encapsulation (Data Protection)
Encapsulation wraps data (attributes) and behavior (methods) into a single unit, and restricts direct access to prevent accidental tampering.
- **Real-World Example:** A player's health points (`hp`) are encapsulated inside their character. Outer code cannot directly change the health points to a negative number or bypass logic. The only way to modify HP is through predefined, validated actions like `take_damage()` or `use_medkit()`.

### 3. Inheritance (Code Reuse)
Inheritance allows a new class (subclass) to inherit attributes and methods from an existing class (parent class), avoiding redundant code.
- **Real-World Example:** You define a base `Character` class with generic properties like `run()` and `jump()`. Then, you create specialized classes like `Alok` and `Kelly` that inherit those behaviors automatically, but add their own unique passive/active abilities.

### 4. Polymorphism (Many Forms)
Polymorphism allows the same method name to perform different actions depending on the object calling it.
- **Real-World Example:** If you call the method `.activate_skill()` on `Kelly`, she gains a temporary speed boost. If you call `.activate_skill()` on `Alok`, he generates a healing aura. The method name is identical, but the execution differs.

---

## 4. Advantages and Disadvantages of OOP

### Advantages
1. **Modularity:** Code is divided into self-contained objects, making it easier to debug, test, and maintain.
2. **Reusability:** Inheritance allows you to reuse code without rewriting it (DRY principle: Don't Repeat Yourself).
3. **Flexibility:** Polymorphism allows a single interface to control different behaviors.
4. **Data Security:** Encapsulation protects variables from unauthorized outside modifications.

### Disadvantages
1. **Size:** OOP programs are often larger and contain more lines of boilerplate code than procedural ones.
2. **Execution Speed:** Due to object allocation and method lookup overhead, OOP programs can be slightly slower than procedural programs, though this is negligible in modern systems.
3. **Design Effort:** Planning an object-oriented system requires careful design up front, which can take more time.

## 5. Classes vs. Objects (In Detail)

Let's define the two fundamental building blocks:

- **Class:** A blueprint, template, or prototype. It defines what attributes and methods the objects created from it will have. It does not occupy memory space for data itself.
- **Object:** A concrete instance of a class. It contains actual values and occupies memory.

```text
    CLASS (Blueprint)                 OBJECTS (Instances)
   ┌─────────────────┐               ┌─────────────────┐
   │  GameCharacter  │               │   player_one    │
   ├─────────────────┤  Instantiates ├─────────────────┤
   │ - username      │ ------------> │ username: "Zed" │
   │ - hp            │               │ hp: 100         │
   ├─────────────────┤               └─────────────────┘
   │ + take_damage() │               ┌─────────────────┐
   │ + heal()        │               │   player_two    │
   └─────────────────┘               ├─────────────────┤
                                     │ username: "Max" │
                                     │ hp: 85          │
                                     └─────────────────┘
```

### Real-World App Example: Instagram User Login
To see how a single Class can spawn multiple independent Objects (instances) in a web application context, let's look at how Instagram handles user login:

**Input:**
```python
class Instagram:
    # A method to log in and greet the user
    def login(self, username, password):
        self.username = username
        print(f"{self.username}, Welcome to Instagram!")

# Creating separate user instances from the Instagram class
kavya = Instagram()
kavya.login('kavya', 1234)

rajitha = Instagram()
rajitha.login('rajitha', 1235)

sreedhar = Instagram()
sreedhar.login('sreedhar', 1236)

chinnu = Instagram()
chinnu.login('chinnu', 1237)
```
**Output:**
```text
kavya, Welcome to Instagram!
rajitha, Welcome to Instagram!
sreedhar, Welcome to Instagram!
chinnu, Welcome to Instagram!
```
*(Notice how each object holds its own `self.username` variable. Defining a single class allows us to manage millions of user logins cleanly!)*

### Defining and Using a Class in Python
In Python, we use the `class` keyword followed by the class name (by convention, using CamelCase like `FreeFireCharacter`). Let's build a fully functioning class with attributes and methods:

**Input:**
```python
class FreeFireCharacter:
    # The Constructor (__init__) initializes individual object data
    def __init__(self, name, special_skill, hp=200):
        self.name = name                 # Instance variable
        self.special_skill = special_skill  # Instance variable
        self.hp = hp                     # Instance variable
        self.is_alive = True             # Instance variable

    # Instance method: Character takes damage
    def take_damage(self, amount):
        if not self.is_alive:
            print(f"⚠️ {self.name} is already knocked out!")
            return

        self.hp -= amount
        if self.hp <= 0:
            self.hp = 0
            self.is_alive = False
            print(f"💀 {self.name} has been knocked out (HP: 0)!")
        else:
            print(f"💥 {self.name} took {amount} damage! Remaining HP: {self.hp}")

    # Instance method: Character heals
    def heal(self, amount):
        if not self.is_alive:
            print(f"❌ Cannot heal! {self.name} is knocked out.")
            return

        self.hp = min(200, self.hp + amount)
        print(f"💚 {self.name} healed by {amount}! Current HP: {self.hp}")

# 1. Instantiate (create) objects from our class blueprint
alok = FreeFireCharacter("Alok", "Drop the Beat")
kelly = FreeFireCharacter("Kelly", "Dash", hp=180)

# 2. Access variables and simulate in-game combat
print(f"Match Start: {alok.name} (Skill: {alok.special_skill}) vs. {kelly.name} (Skill: {kelly.special_skill})")

alok.take_damage(80)
kelly.take_damage(190)
alok.heal(50)
kelly.heal(10)  # Kelly is knocked out, so she cannot heal!
```
**Output:**
```text
Match Start: Alok (Skill: Drop the Beat) vs. Kelly (Skill: Dash)
💥 Alok took 80 damage! Remaining HP: 120
💀 Kelly has been knocked out (HP: 0)!
💚 Alok healed by 50! Current HP: 170
❌ Cannot heal! Kelly is knocked out.
```
