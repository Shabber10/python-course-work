# 1. Introduction to OOP (Classes and Objects)

Object-Oriented Programming (OOP) is one of the most powerful and widely used programming paradigms in modern software development. Instead of writing code as a series of sequential instructions, OOP allows you to structure your programs by modeling real-world entities and their behaviors.

---

## 1. What is OOP?

Think about how you use a streaming app like **Netflix**. When you open the app, you see multiple **Profiles** under your account: one for you, one for your sibling, and one for your parents. 

Each profile has:
- **State (Attributes/Data):** Profile name, language preference, maturity rating, and watch history.
- **Behaviors (Methods/Actions):** Playing a video, adding a show to "My List", and changing subtitle settings.

In programming terms:
- The general layout or template of what a profile is and does is called a **Class**.
- Each individual active profile is an **Object** (or an **Instance** of that Class).

Object-Oriented Programming is a method of designing programs by grouping related variables (state) and functions (behavior) into single entities called **Objects**.

---

## 2. Why Do We Need OOP? (Procedural vs. OOP)

To see why OOP is so valuable, let’s look at how we would build a **Bank Account Management System** using **Procedural Programming** (the style we have been using so far) versus **Object-Oriented Programming**.

### A. The Procedural Approach (Loose Data)
In procedural programming, data is separated from functions. We might track accounts using parallel lists:

**Input:**
```python
account_owners = ["Alice", "Bob", "Charlie"]
account_balances = [5000, 12000, 800]
account_numbers = [101, 102, 103]

def withdraw(account_index, amount):
    if account_balances[account_index] >= amount:
        account_balances[account_index] -= amount
        print(f"✅ Withdrew ${amount} from {account_owners[account_index]}'s account. Remaining: ${account_balances[account_index]}")
    else:
        print(f"❌ Insufficient funds for {account_owners[account_index]}!")

# Simulate withdrawal
withdraw(0, 1000)
```
**Output:**
```text
✅ Withdrew $1000 from Alice's account. Remaining: $4000
```

> [!CAUTION]
> **Why this fails in real systems:**
> - If Charlie closes his account, you must delete his data from *every single list* and manually adjust all indices. If you make a mistake, Alice might inherit Charlie's money!
> - The data is completely unprotected. Any part of the program can write `account_balances[0] = -5000` directly, bypassing the withdrawal check.

### B. The OOP Approach (Bundled Entity)
In OOP, we bind the customer's account data and actions together. If an account is closed, we simply delete that single `BankAccount` object.

**Input:**
```python
class BankAccount:
    def __init__(self, owner, account_number, balance=0):
        self.owner = owner
        self.account_number = account_number
        self.balance = balance

    def withdraw(self, amount):
        if self.balance >= amount:
            self.balance -= amount
            print(f"✅ Withdrew ${amount} from {self.owner}'s account. Remaining: ${self.balance}")
        else:
            print(f"❌ Insufficient funds for {self.owner}!")

# Create accounts (Objects)
acct1 = BankAccount("Alice", 101, 5000)
acct2 = BankAccount("Bob", 102, 12000)

# Simulate actions directly on the objects
acct1.withdraw(1000)
```
**Output:**
```text
✅ Withdrew $1000 from Alice's account. Remaining: $4000
```

## 3. The Four Pillars of OOP

OOP is defined by four core principles (features) that make software modular, clean, and extensible.

```text
       ┌─────────────────────────────────────────────────────────┐
       │                 THE FOUR PILLARS OF OOP                 │
       └─────────────────────────────────────────────────────────┘
            │               │               │               │
     ┌──────┴──────┐ ┌──────┴──────┐ ┌──────┴──────┐ ┌──────┴──────┐
     │ Abstraction │ │Encapsulation│ │ Inheritance │ │Polymorphism │
     └─────────────┘ └─────────────┘ └─────────────┘ └─────────────┘
```

### 1. Abstraction (Hiding Complexity)
Abstraction hides complex background details and only exposes the essential controls.
- **Real-World Example:** In a game like *Free Fire*, you don't need to know the intricate physics, server ping calculations, or hitbox rendering algorithms required to shoot a weapon. You simply press the "fire" button and the game handles the complex logic automatically.

### 2. Encapsulation (Data Protection)
Encapsulation wraps data (attributes) and behavior (methods) into a single unit, and restricts direct access to prevent accidental tampering.
- **Real-World Example:** In a bank account system, a customer's `balance` is encapsulated. Other code cannot write `acct.balance = -9999` directly. The balance can only be modified through predefined, validated methods like `deposit()` or `withdraw()` which enforce safety rules.

### 3. Inheritance (Code Reuse)
Inheritance allows a new class (subclass) to inherit attributes and methods from an existing class (parent class), avoiding redundant code.
- **Real-World Example:** Imagine a **Smart Home** system. You define a base class `SmartDevice` with common properties like `power_status` and actions like `turn_on()` / `turn_off()`. Then, you create subclasses like `SmartLight` (which inherits power controls but adds `set_brightness()`) and `SmartThermostat` (which adds `set_temperature()`).

### 4. Polymorphism (Many Forms)
Polymorphism allows the same method name to perform different actions depending on the object calling it.
- **Real-World Example:** Think of a **Notification System** in a social media app. You have a method called `send_notification()`. If it is called on an `EmailNotification` object, it format-renders and sends an email. If it is called on an `SMSNotification` object, it sends a text message. The command name is identical, but the execution differs.

---

## 4. Advantages and Disadvantages of OOP

### Advantages
1. **Modularity:** Code is divided into self-contained objects, making it easier to debug, test, and maintain.
2. **Reusability:** Inheritance allows you to reuse code without rewriting it (DRY principle: Don't Repeat Yourself).
3. **Flexibility:** Polymorphism allows a single interface to control different behaviors.
4. **Data Security:** Encapsulation protects variables from unauthorized outside modifications.

### Disadvantages
1. **Size:** OOP programs are often larger and contain more lines of boilerplate code than procedural ones.
2. **Execution Speed:** Due to object allocation and method lookup overhead, OOP programs can be slightly slower than procedural programs, though this is negligible in modern systems.
3. **Design Effort:** Planning an object-oriented system requires careful design up front, which can take more time.

## 5. Classes vs. Objects (In Detail)

Let's define the two fundamental building blocks:

- **Class:** A blueprint, template, or prototype. It defines what attributes and methods the objects created from it will have. It does not occupy memory space for data itself.
- **Object:** A concrete instance of a class. It contains actual values and occupies memory.

```text
       CLASS (Blueprint)                 OBJECTS (Instances)
     ┌──────────────────┐               ┌──────────────────┐
     │   YouTubeVideo   │               │    video_one     │
     ├──────────────────┤  Instantiates ├──────────────────┤
     │ - title          │ ------------> │ title: "Learn Python"
     │ - views          │               │ views: 1520      │
     ├──────────────────┤               └──────────────────┘
     │ + play()         │               ┌──────────────────┐
     │ + like()         │               │    video_two     │
     └──────────────────┘               ├──────────────────┤
                                        │ title: "OOP Basics"
                                        │ views: 420       │
                                        └──────────────────┘
```

### Real-World App Example: Instagram User Login
To see how a single Class can spawn multiple independent Objects (instances) in a web application context, let's look at how Instagram handles user login:

**Input:**
```python
class Instagram:
    # A method to log in and greet the user
    def login(self, username, password):
        self.username = username
        print(f"{self.username}, Welcome to Instagram!")

# Creating separate user instances from the Instagram class
user1 = Instagram()
user1.login('john', 1234)

user2 = Instagram()
user2.login('emma', 1235)

user3 = Instagram()
user3.login('sam', 1236)

user4 = Instagram()
user4.login('lisa', 1237)
```
**Output:**
```text
john, Welcome to Instagram!
emma, Welcome to Instagram!
sam, Welcome to Instagram!
lisa, Welcome to Instagram!
```
*(Notice how each object holds its own `self.username` variable. Defining a single class allows us to manage millions of user logins cleanly!)*

### Defining and Using a Class in Python
In Python, we use the `class` keyword followed by the class name (by convention, using CamelCase like `YouTubeVideo`). Let's build a fully functioning class with attributes and methods:

**Input:**
```python
class YouTubeVideo:
    # The Constructor (__init__) initializes individual object data
    def __init__(self, title, creator, duration_sec):
        self.title = title                 # Instance variable
        self.creator = creator             # Instance variable
        self.duration_sec = duration_sec   # Instance variable
        self.views = 0                     # Default instance variable
        self.likes = 0                     # Default instance variable

    # Instance method: Simulate watching the video
    def play(self):
        self.views += 1
        print(f"▶️ Watching '{self.title}' by {self.creator}. Views: {self.views}")

    # Instance method: Simulate liking the video
    def like(self):
        self.likes += 1
        print(f"👍 You liked '{self.title}'! Total Likes: {self.likes}")

# 1. Instantiate (create) objects from our class blueprint
video1 = YouTubeVideo("Python OOP for Beginners", "TechEdu", 600)
video2 = YouTubeVideo("Top 5 Python Tricks", "CodeCraft", 300)

# 2. Access variables and simulate behavior
print(f"Uploaded: '{video1.title}' by {video1.creator} ({video1.duration_sec}s)")

video1.play()
video1.play()
video1.like()

video2.play()
```
**Output:**
```text
Uploaded: 'Python OOP for Beginners' by TechEdu (600s)
▶️ Watching 'Python OOP for Beginners' by TechEdu. Views: 1
▶️ Watching 'Python OOP for Beginners' by TechEdu. Views: 2
👍 You liked 'Python OOP for Beginners'! Total Likes: 1
▶️ Watching 'Top 5 Python Tricks' by CodeCraft. Views: 1
```
